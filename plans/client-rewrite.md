# Client rewrite: de-Google-ing the dependency tree

Status: written 2026-07-03. Tracking issue: #39. Target: 0.4/1.0 — explicitly
**not** coupled to 0.3.

## Goal

Drop `google_api_storage`, `google_gax` (and transitively Poison and Tesla),
and the legacy-Goth coupling. Replace with a few hundred lines of hand-written
client that we own, understand, and can debug.

**Decision: no discovery-based codegen.** Generating a client from Google's API
discovery documents is exactly how `google_api_storage`/`google_gax` became the
unmaintained pile we're escaping — machine-written code nobody reads,
regenerated by nobody, dragging Poison behind it. The GCS JSON API surface this
adapter needs has been stable for a decade and fits in one module:

| Operation | Used by |
|---|---|
| `objects.insert` (multipart) | `put/3` today |
| `objects.insert` (**resumable**) | large uploads — fixes the current whole-file-in-RAM buffering |
| `objects.get` | metadata assertions, repair tooling |
| `objects.delete` | `delete/3` |
| `objects.list` (with `prefix`) | test cleanup (#35), repair tooling (#40) |
| `objects.rewrite` | repair tooling (#40) — server-side copy, no re-upload |
| IAM Credentials `signBlob` | signed URLs without a local private key (workload identity / Cloud Run — see #18) |

## Prerequisites (hard gates)

1. **#32 — library-owned result types.** Consumers currently pattern-match
   `{:error, %Tesla.Env{}}` and `%GoogleApi...Object{}`. Normalize in 0.3 so
   this rewrite changes nothing consumers see.
2. **#38 — offline contract suite.** The feature suite, dual-targeted at
   fake-gcs-server, is the frozen spec. The rewrite is done when the same suite
   passes against both targets with the new client.

## Architecture: three swappable seams

The organizing principle: every place we currently depend on someone else's
abandoned abstraction becomes a seam we own, with a behaviour on one side and
pure functions on the other.

### 1. Transport (the "swappable network layer")

- **Pure request builders**: functions taking (definition-derived data) and
  returning a request as *data* — `%Request{method, url, headers, query, body}`.
  Object naming, acl/predefinedAcl mapping (#27), header merging,
  multipart-vs-iodata-vs-resumable choice all live here, unit-tested with no
  network. (Started during #38 — the rewrite inherits those tests.)
- **`Transport` behaviour**: `execute(request, opts) :: {:ok, %Response{}} | {:error, %Error{}}`.
  One default implementation on a maintained HTTP client. Candidates: Req
  (best DX, brings Finch), Finch directly (leaner), or hackney (already a
  waffle dependency — zero new deps). Decide at implementation time; the
  behaviour makes it a contained decision. Nothing outside the transport module
  may reference the HTTP library.
- Consumers get a documented escape hatch: configure a custom transport
  (proxies, instrumentation, exotic runtimes) without forking.

### 2. Signer as an adapter (not a URL-builder appendage)

The v4 signer (#14) is a standalone module behind a behaviour — input:
credentials + canonical request parts; output: signed URL/headers. Not welded
into `UrlV2`-style URL assembly the way v2 signing is today.

Why it matters beyond this repo: **GCS v4 signing is essentially AWS Signature
V4 with different header names and credential scope.** A cleanly separated
SigV4-style signer:

- makes the v2 → v4 migration testable in isolation (Google publishes v4 test
  vectors; add them next to our own fixed-PEM vectors from #37),
- leaves the door open for upstream Waffle to split S3 signing out of `ex_aws`
  and share a signer implementation,
- gives `signBlob`-backed signing (keyless environments) a natural place: it's
  just another implementation of the signing behaviour where the RSA operation
  is delegated to the IAM API instead of `:public_key.sign/3`.

v2 signing ships one last time in 0.3 (with #29 fixed), is deprecated in the
rewrite, and removed when Google turns it off or 1.0, whichever first.

### 3. Tokens

Keep the `Token.Fetcher` behaviour as the single seam (per the #34 decision in
0.3). The rewrite extends it so **signing credentials flow through the same
abstraction** — `client_email` + private key, or a signBlob delegate — killing
the last hardcoded `Goth.Config` reads. Goth (modern API, supervised, named
server) is the default implementation; it stops being a load-bearing wall. If
someone wants gcloud-CLI creds, metadata-server-only, or a Vault-backed
fetcher, it's one behaviour implementation, not a fork.

## Non-goals

- General GCP client. This is `waffle_gcs`'s client, sized to `waffle_gcs`.
- Preserving `conn/1` returning a `Tesla.Env.client()` — that type disappears;
  #32's normalized contract is what's preserved.

## Done when

- `mix deps.tree` shows no `google_api_storage`, `google_gax`, `poison`,
  `tesla`.
- The dual-target feature suite passes unchanged (modulo `:live_only` tags).
- Resumable upload works for a file larger than available RAM (streamed).
- Signed URLs work in a keyless (workload identity) environment via signBlob.
